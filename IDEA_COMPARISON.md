# RocksDB时序数据库优化方案对比

## 一、方案概述

| 方案 | 定位 | 设计理念 | 适用场景 |
|------|------|----------|----------|
| **kimi25** | 极简轻量 | 代码极简、零依赖、易用优先 | IoT设备、边缘计算、原型验证 |
| **minimax25** | 轻量高性能 | 单文件、内存为主、持久化 | 中小规模、GB级数据 |
| **glm5** | 生产级 | 模块化、功能完整、可扩展 | 大规模生产、TB级数据 |

## 三、存储层对比

### 3.1 Key设计

| 方案 | Key格式 | Key大小 | 特点 |
|------|---------|---------|------|
| **kimi25** | 二进制紧凑Key | ~20字节 | 最小化存储 |
| **minimax25** | 字符串Key | ~40字节 | 可读性好 |
| **glm5** | 块级复合Key | ~30字节 | 块内聚合 |

**Key结构对比：**

```c
// kimi25: 极简二进制Key (~20字节)
struct {
    uint8_t  type;        // 'D'
    uint16_t meas_len;    // measurement长度
    char     measurement[]; // 变长
    uint64_t series_id;   // 8字节
    uint64_t timestamp;   // 8字节（秒级）
} compact_key_t;

// minimax25: 标准字符串Key (~40字节)
"D:cpu_usage:123456789:1700000000"

// glm5: 块级Key (~30字节)
struct {
    uint8_t  prefix;      // 'D'
    uint8_t  meas_len;
    char     measurement[];
    uint64_t series_id;
    uint32_t block_ts;    // 块起始秒
    uint16_t block_offset; // 块内偏移
} block_key_t;
```

### 3.2 存储机制

| 特性 | kimi25 | minimax25 | glm5 |
|------|--------|-----------|------|
| 存储粒度 | 单点 | 时间块(60秒) | 时间块(30秒) |
| Value格式 | 字符串 | 二进制数组 | 二进制+Qualifier |
| 压缩方式 | LZ4 | Delta+LZ4 | Gorilla+ZSTD |
| 压缩率 | 2:1 | 5:1 | 10:1~20:1 |

### 3.3 冷热分离

| 方案 | 实现方式 | 删除过期数据 |
|------|----------|--------------|
| **kimi25** | 无 | 遍历删除 |
| **minimax25** | ColumnFamily按日分 | Drop CF |
| **glm5** | ColumnFamily + 压缩策略 | Drop CF |

**glm5冷热分离架构：**
```
┌─────────────────────────────────────────────────────────┐
│                    RocksDB Instance                      │
├─────────────┬─────────────┬─────────────┬───────────────┤
│ CF:meta     │ CF:20240101 │ CF:20240102 │ CF:20240103   │
│ (元数据)    │ (冷数据)    │ (温数据)    │ (热数据)      │
│             │ ZSTD压缩    │ LZ4压缩     │ 无压缩        │
│             │ 无Compaction│ 自动Compact │ 自动Compact   │
└─────────────┴─────────────┴─────────────┴───────────────┘
```

## 四、写入优化对比

### 4.1 批量写入

| 方案 | 批量策略 | 后台线程 | WAL控制 |
|------|----------|----------|---------|
| **kimi25** | 自动缓冲(1000点) | 有 | 可开关 |
| **minimax25** | 自动缓冲(5000点) | 有 | 可开关 |
| **glm5** | 手动+自动 | 有 | 可开关 |

**kimi25批量写入：**
```c
typedef struct {
    point_t* buffer;
    size_t count;
    size_t capacity;
    size_t batch_size;        // 1000
    uint64_t flush_interval_ms; // 100ms
    pthread_t flush_thread;

```

### 4.2 写入性能

| 方案 | 单点写入 | 批量写入 | 优化手段 |
|------|----------|----------|----------|
| **kimi25** | 50K/s | 300K/s | 无WAL + 自动批量 |
| **minimax25** | 50K/s | 200K/s | 块聚合 + 批量 |
| **glm5** | 50K/s | 500K/s | Gorilla压缩 + 批量 |

## 五、查询优化对比

### 5.1 查询机制

| 特性 | kimi25 | minimax25 | glm5 |
|------|--------|-----------|------|
| 扫描方式 | 前缀扫描 | 块扫描+剪枝 | 块扫描+索引 |
| 范围限制 | 迭代器边界 | 时间范围剪枝 | 时间范围+索引 |
| 并行查询 | 无 | 多线程 | 多线程 |
| 结果缓存 | 简单LRU | 多级缓存 | 复杂缓存 |

### 5.2 查询性能

| 方案 | 点查询 | 范围查询 | 聚合查询 | 优化手段 |
|------|--------|----------|----------|----------|
| **kimi25** | 1ms | 20ms | 50ms | 前缀+缓存 |
| **minimax25** | 1ms | 10ms | 30ms | 块剪枝+并行 |
| **glm5** | 0.5ms | 5ms | 10ms | 索引+向量化 |

### 5.3 缓存策略

```
kimi25: 简单LRU缓存（固定1024大小）
┌─────────────────────────────────────┐
│  Cache Entry[1024]                  │
│  ├─ key_hash                        │
│  ├─ result (复制)                   │
│  ├─ timestamp                       │
│  └─ hit_count                       │
└─────────────────────────────────────┘

minimax25: 多级缓存
┌─────────────────────────────────────┐
│  查询结果缓存 (LRU, 大容量)          │
├─────────────────────────────────────┤
│  块数据缓存 (LRU, 中容量)            │
├─────────────────────────────────────┤
│  元数据缓存 (Hash, 热点数据)         │
└─────────────────────────────────────┘

glm5: 复杂缓存系统
┌─────────────────────────────────────┐
│  查询结果缓存 (带过期时间)            │
├─────────────────────────────────────┤
│  块数据缓存 (LRU + 预加载)           │
├─────────────────────────────────────┤
│  索引缓存 (Bloom Filter加速)         │
├─────────────────────────────────────┤
│  RocksDB块缓存 (配置可调)            │
└─────────────────────────────────────┘
```

| 方案 | 索引类型 | 存储位置 | 更新方式 | 适用场景 |
|------|----------|----------|----------|----------|

rocksdb_filterpolicy_t* bloom_filter;

## 七、压缩对比

| 方案 | 压缩算法 | 压缩率 | CPU开销 | 适用数据 |
|------|----------|--------|---------|----------|
| **kimi25** | LZ4 | 2:1 | 低 | 热数据 |
| **minimax25** | Delta + LZ4 | 5:1 | 中 | 温数据 |
| **glm5** | Gorilla + ZSTD | 10:1~20:1 | 高 | 冷数据 |

**压缩算法对比：**

```
原始数据: 1000个数据点
├─ 时间戳: 8字节 × 1000 = 8000字节
└─ 数值:   8字节 × 1000 = 8000字节
   总计: 16000字节

kimi25 (LZ4):
└─ 压缩后: ~8000字节 (2:1)

minimax25 (Delta + LZ4):
├─ 时间戳Delta: 1-4字节 × 1000 = ~2000字节
├─ 数值XOR:     2-8字节 × 1000 = ~4000字节
└─ 压缩后: ~3000字节 (5:1)

glm5 (Gorilla + ZSTD):
├─ Gorilla压缩: ~1500字节
└─ ZSTD压缩后:  ~800字节 (20:1)
```

## 十二、总结建议

### 12.1 选择决策树

```
                    开始
                     │
            数据规模 < 1GB ?
            /           \
          是              否
          │               │
    选择kimi25    需要生产级功能?
                  /           \
                是              否
                │               │
          选择glm5      选择minimax25
```

### 12.2 推荐方案

| 优先级 | 场景 | 方案 | ROI |
|--------|------|------|-----|
| 1 | 快速验证 | kimi25 | 最高 |
| 2 | 中小生产 | minimax25 | 高 |
| 3 | 大规模生产 | glm5 | 高 |

### 12.3 演进路径

```
kimi25 (原型验证)
    ↓ 需求增长
minimax25 (中小规模)
    ↓ 规模扩大
glm5 (大规模生产)
```

三个方案形成**渐进式演进**的技术栈，可根据业务发展阶段平滑升级。
